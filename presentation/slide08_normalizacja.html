Normalization vs denormalization
The right structure for the right job: OLTP, analytics, and archive
10 / 18 Polyglot persistence
GiftFlow
Operational layer
3NF — Azure SQL
Read/write of requests and decisions (OLTP)
request → FK employee, beneficiary, gift_type
approval → FK request, approver
reference tables without redundancy
Integrity, no anomalies, consistent transactional writes.
Easy rule enforcement (CHECK, FK, uniqueness).
Many JOINs — worse for ad-hoc reports.
Analytical layer
Star schema — Fabric
Reports: department × month × gift type
Fast aggregations, simple for Power BI / DAX.
Redundancy in dimensions (deliberate, controlled).
Audit & archive
Denormalization / JSON
Immutable events (append-only, WORM)
{ "event": "APPROVED",
  "request": {...snapshot...},
  "actor": "director@...",
  "ts": "2026-06-17T20:10Z" }
Full context in one document — indisputable trail.
Cheap writes (Blob/Cosmos), schema-on-read.
No consistency enforcement — read/audit only.
OLTP → 3NF (Azure SQL): integrity and rules BI → star schema (Fabric): efficient reports Audit → JSON append-only (Blob WORM / Cosmos): immutability
